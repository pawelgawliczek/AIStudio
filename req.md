# MCP Control Plane / AI Agentic Studio

## 1. Vision

A unified control plane built as an MCP server that:

* Manages AI agentic frameworks across tools like Claude Code / Claude Desktop / Codex.
* Acts as single source of truth for projects, workflows, agents, and metrics.
* Standardizes workflows and enables one-command bootstrap for new repos.
* Compares effectiveness of different agentic setups (not humans vs AI).

## 2. Key Capabilities

### 2.1 Project & Planning Layer

* Enforce clear hierarchy:

  * **Epic** → high-level initiative.
  * **Story** → vertical slice within an epic.
  * **Subtask** → smallest executable unit, primary target for agents.
* Agents operate only on **subtasks**; humans reason mainly at story/epic level.
* All metrics and artifacts roll up: Subtask → Story → Epic.
* Maintain canonical project model:

  * Projects, epics, stories/tasks, priorities, labels.
  * Explicit linkage between work items and agents/humans.
* Provide a web UI for PMs to:

  * Create/modify epics and stories.
  * Set priorities and status.
  * Assign ownership to agents or humans.
  * Control which frameworks/agents are allowed to work on which stories.

### 2.2 Agentic Framework Management

* Store and version:

  * Agent definitions (prompts, roles, tools, constraints, safety rules).
  * Agent frameworks: bundles of agents + routing/policies.
* Allow configuration via web UI ("Agent Studio"):

  * Visual or structured editing of agents and frameworks.
  * Activate/deactivate frameworks per project.
  * Generate host-specific config (Claude, Codex, etc.).
* Auto-generate skill/agent files for supported hosts:

  * For Claude Code: `.claude/` skills, CLAUDE.md, etc.
  * For other clients: their required layout.

### 2.3 MCP Server Behavior

* Implement as MCP server that:

  * Exposes tools for project, story, artifact, agent, framework, and metrics operations.
  * Uses `clientInfo` from MCP `initialize` to detect host (Claude vs Codex vs other) and tailor responses.
* Core tools (examples):

  * `bootstrap_project(...)` – create project, default backlog, agents, frameworks, and return file plan + post-actions.
  * `list_projects`, `list_epics`, `list_stories`, `get_story`, `update_story`, `create_story`.
  * `list_agents`, `get_agent`, `update_agent`, `list_frameworks`, `get_framework`, `update_framework`, `activate_framework`.
  * `list_artifacts`, `link_artifact`.
  * `get_changes(project_id, since_cursor)` – sync delta for hosts.
  * `log_run(payload)` – store telemetry from each agent action.
  * `get_agent_metrics(agent_id/framework_id, window)` – aggregated KPIs.

### 2.4 One-Command Bootstrap Flow

* Inside Claude Code (or other MCP host), user issues natural command:

  * e.g. "Initialize AI Studio for this repo".
* System/skill prompt instructs host to:

  * Call `bootstrap_project` on MCP server with repo metadata.
  * Apply returned file plan (create skill files, CLAUDE.md, agent config).
  * Optionally run safe post-actions.
* Result:

  * Repo wired to control plane.
  * Metrics and workflows configured from start.

## 3. Metrics & KPIs (Agentic Setup Effectiveness)

Focus: compare different **agentic workflows** (e.g. Dev-only vs Dev+Architect vs Dev+Architect+Tester), not humans vs AI.

### 3.1 Tracked Entities

* `Run` records with:

  * project_id, story_id, agent_id, framework_id.
  * origin (Claude Code, CLI, CI, etc.).
  * timestamps (start/finish), success flag, error type.
  * tokens_in, tokens_out.
  * iterations count for that work item.
* Links to:

  * Code artifacts (PRs, commits).
  * Defects/bugs.
  * Static analysis outputs.

### 3.2 KPIs

Per workflow / framework / agent:

* Time to complete story (assignment → done).
* Token usage per story and per agent.
* Number of iterations/prompts per completed story.
* Code churn: rewrites after initial agent output.
* Code complexity deltas (e.g. cyclomatic complexity).
* Defects per story and severity (linked back to workflow variant).
* Defect leakage: prod vs pre-prod for agent-produced work.
* Human override rate and escalation events.
* Throughput: completed stories per time window by workflow.
* Cost per story (token + review time modelled).
* Relative efficiency: compare frameworks against each other.

## 4. Host-Specific Behavior

* On `initialize`, capture `clientInfo`:

  * Adjust `bootstrap_project` outputs and config:

    * Claude Code: output `.claude/skills/*`, CLAUDE.md, etc.
    * (Future) Codex/other: generate their native config.
* Same control plane, multiple client profiles.

## 5. Integration Points

* MCP tools as primary integration into Claude Code / other MCP-aware IDEs.
* CLI for terminals/CI:

  * Read `project_id` from repo config generated by `bootstrap_project`.
  * Log runs, link artifacts and defects.
* Web UI:

  * Planning (epics/stories).
  * Agent/Framework management.
  * Metrics dashboards.

## 6. Non-Requirements / Constraints

* No direct server-initiated execution inside Claude: all flows are pull-based via MCP tools.
* Focus on:

  * Deterministic structures.
  * Transparent actions.
  * Vendor-agnostic where possible (Claude, Codex, others).
* Designed to be self-hostable and also viable as SaaS.

## 7. Sync Model & Live Execution View

* On each CLI/Claude Code command, MCP server:

  * Identifies project/session.
  * Checks for changes since last sync (stories, priorities, agents, frameworks, artifacts).
  * Returns a concise summary of changes and prompts user to sync.
* `get_changes(project_id, since_cursor)`:

  * Returns delta set: updated/created/deleted epics, stories, agents, frameworks, artifacts.
  * Designed to be fast and minimal.
* Sync flow in Claude Code:

  * If changes exist, suggest: "First sync project context".
  * On confirmation, apply updates to local files/config/context.
  * Show human-readable summary of key changes (e.g. new priorities, reassigned stories, new constraints).
* All user actions via Claude Code/CLI (creating docs, running tasks, updating stories) call MCP tools (`log_run`, `update_story`, `link_artifact`) so state is immediately reflected in the web UI.
* Live view:

  * Web UI subscribes (WS/long-poll) to run/events stream.
  * Shows in-flight runs: which agents are active, which tasks executing, their status and progress.

## 8. Planning, Complexity & Architectural Support

* Each story, after review by BA and Architect, must carry structured assessments:

  * `business_impact` (e.g. 1–5 or LOW/MED/HIGH): effect on revenue, risk, user value.
  * `business_complexity` (1–5): stakeholder alignment, process changes, edge cases.
  * `technical_complexity` (1–5): code/architecture impact, integrations, risk, unknowns.

* These fields are mandatory before assignment to agents and are stored on the story entity.

* Over time this enables baselining:

  * "Easy" vs "hard" stories per dimension.
  * Performance of each agent/framework segmented by complexity band.
  * Example: compare Architect+Dev vs Dev-only only within same complexity ranges.

* Planning is first-class and structured:

  * Support professional backlog management: epics, dependency mapping, milestones.
  * Allow grouping/combining stories where Architect agents identify shared dependencies.

* Architect agent workflow:

  * Given a plan and complexity data, use MCP tools to:

    * Analyze dependencies across stories.
    * Propose merges/splits of stories.
    * Suggest implementation order and ownership (which agents/humans).
    * Flag architectural or cross-cutting concerns.
  * Suggestions surfaced in web UI for PM/tech lead approval, then applied via MCP.

## 9. Code / Commit Linking

* Every commit must be traceable to planned work.
* Enforced convention:

  * Each commit message and/or branch name includes a valid `STORY-XXX` ID.
  * A Git hook/CI check can reject commits/PRs without a valid mapping.
* MCP server exposes `link_commit({ commit_hash, story_id })` which:

  * Persists linkage to story and its parent epic.
  * Records author, timestamp, and repo.
  * Optionally stores LOC added/removed, files touched, and complexity deltas.
* Data model:

  * `commits` table or artifacts including:

    * `commit_hash`, `project_id`, `story_id`, `epic_id`, `author`, `timestamp`.
* This enables per-story and per-epic:

  * Exact view of implemented code.
  * Code quality metrics tied directly to planned work.
  * Reliable inputs for refactor suggestions and agent performance KPIs.

## 10. Business Analysis & Documentation

* The Business Analyst (BA) role maintains living project documentation.
* After each feature's implementation and approval:

  * Automatically or manually generate entries in a **Functionality Catalogue**:

    * feature name, story/epic references, description, status, owner.
    * linked commits, artifacts, metrics.
  * Maintain a **Use Case Library**:

    * real business scenarios, actors, flows, dependencies.
    * traceable back to features and stories.
* All stored in the MCP database and exposed via web UI for quick navigation.
* BA responsibilities:

  * Ensure catalogue and use-case docs are current and complete.
  * Add contextual business notes, KPIs, screenshots, and acceptance criteria.
* To improve retrieval:

  * Enable semantic search/RAG layer over this documentation (e.g. via vector index).
  * Let BAs or agents query: "What features handle user registration?" and get precise, linked answers.
* This creates always-up-to-date project specification accessible to PMs, BAs, Architects, and agents.

## 11. Cost Estimation & Token-based Complexity

* During Planning (Phase 1), PM must estimate for each story:

  * `business_impact`
  * `business_complexity`
  * `technical_complexity`
  * `estimated_token_cost` or `ai_effort` bucket (XS/S/M/L/XL).
* Stories without these fields set cannot progress to BA/Architect phases.
* After completion, system compares:

  * estimated vs actual token usage,
  * estimated vs actual AI interaction cycles.
* Uses:

  * Better future cost prediction.
  * Additional calibrated complexity marker for comparing agentic setups.

## 12. Architect Support Tools & Code KPIs

* Background metric collection:

  * MCP control plane continuously gathers code quality KPIs without interrupting dev/agent workflows.
  * Data sources:

    * Git diffs per linked commit.
    * Static analysis (complexity, smells, duplication, security) via OSS tools.
    * Test coverage reports from CI.
    * Churn and hotspot analysis from VCS history.
  * All processing is handled by backend workers; developers and agents are not blocked or forced into extra steps beyond commit conventions.
* Storage & access:

  * Metrics stored per file, component, layer, commit, story, and epic.
  * Exposed via MCP tools (e.g. `get_architect_insights`, `get_hotspots`) so agents can consume them directly.
  * Exposed via web UI dashboards for architects and leads:

    * Hotspots list.
    * Risk indicators per component.
    * Trends over time.
* Architect usage:

  * Use KPIs to decide which files/components to refactor.
  * Use as input during planning and architecture phases to guide decomposition and technical decisions.
  * No manual data wrangling; insights are always up to date and queryable.
* Optionally enable "Architect Insights" for a project.
* This mode provisions additional MCP/HTTP endpoints that compute structural KPIs at multiple levels:

  * **Layers:** frontend, backend, infra, tests, etc.
  * **Components:** e.g. authentication, billing, search, API gateway.
  * **Files:** per-file LOC, churn, complexity, ownership.
* Implementation outline:

  * Maintain a mapping configuration:

    * which paths belong to which layer/component.
  * On demand or on schedule:

    * scan repo diffs or full tree,
    * compute LOC, complexity scores, churn, and hotspots,
    * expose results via MCP tools like `get_architect_insights(project_id)`.
* Architect agents and humans use these metrics to:

  * Identify imbalanced modules and hotspots.
  * Recommend refactors and test coverage improvements.
  * Validate whether current architecture still fits growth and change patterns.

## 13. Installation & Local Setup (MVP)

* Target platforms:

  * macOS, Linux as primary.
  * Windows supported via WSL2 (recommended for v1).
* Repository layout (high level):

  * `install.sh` – single entrypoint.
  * `docker-compose.yml` – MCP server, DB, cache, web UI, optional analyzers.
  * `backend/` – MCP server + API.
  * `frontend/` – web UI.
  * `cli/` – thin `ai-studio` CLI.
  * `migrations/`, `.env.example`, `README.md`.
* `install.sh` responsibilities:

  * Verify Docker and Docker Compose.
  * Copy `.env.example` to `.env` if missing and generate secrets.
  * Run DB migrations.
  * Start stack via `docker compose up -d`.
  * Output:

    * MCP endpoint (e.g. `http://localhost:8765`).
    * Web UI URL (e.g. `http://localhost:8080`).
    * Basic instructions for connecting MCP in Claude/Codex.
* `ai-studio init` (run inside a repo):

  * Detects git + tech stack.
  * Calls `bootstrap_project` on MCP server.
  * Writes `.ai-studio/config.json` with `project_id`.
  * Generates `.claude/` (or host-specific) config/skills files.
  * Prints next steps for enabling MCP in the IDE.

## 14. MVP Scope & Future Extensibility

* This design is the MVP of a more general control plane:

  * Initially focused on software projects with roles:

    * BA: maintains use cases and business specs.
    * Architect: manages architecture, structural metrics, refactor insights.
    * Tester/QA: owns test cases and their mapping to code.
    * Dev agents: implement subtasks with minimal context.
* Future-ready abstractions:

  * "Role blocks" / agent types are generic:

    * Later can add blocks like: GraphicDesigner, MarketingPlanner, ContentWriter, etc.
    * Each block can be granted specific tools, data scopes, and workflows.
  * Workflow engine:

    * Users can compose custom workflows from these role blocks via web UI.
    * Current described flow (BA → Architect → UI → Dev → QA) is a template, not hard-coded.
  * Per-agent customization:

    * Web UI must allow attaching custom instructions/policies per agent/block per project.
    * These configs are stored in the control plane and propagated to generated skill/agent files.
* Goal: same core system can orchestrate domain-specific agent workflows (IT, marketing, design, etc.) without redesigning the platform.

## 15. Defects and Leakage Tracking

* Add new story type: **Defect/Bug**.

  * Each defect must link to the originating story and epic.
  * Store metadata:

    * `origin_stage`: where defect was introduced (e.g. dev, architect, BA).
    * `discovery_stage`: where defect was found (e.g. test, UAT, production).
    * `severity` and `impact`.
  * Enables calculation of **defect leakage** = defects discovered after initial QA ÷ total defects.
  * Automatically updates per framework/agent metrics when defects are logged.

## 16. Agentic Framework Effectiveness KPIs

Framework performance measured through efficiency, quality, and comparability across complexity bands.

### 16.1 Efficiency Metrics

* **Token usage:** input/output tokens per agent and per framework.
* **Token efficiency:** output ÷ input ratio.
* **Estimated vs actual token cost.**
* **Overhead ratio:** (PM + BA + Architect + Tester tokens) ÷ (Dev-agent tokens).
* **Cycle time:** story start → completion.
* **Lead time:** creation → delivery to production.
* **Agent runtime:** execution duration per subtask.
* **Prompt iteration metrics:** prompts per story, rework attempts, prompt-to-success ratio.
* **Parallelization efficiency:** total story time ÷ sum of subtask durations.

### 16.2 Quality Metrics

* **Code Quality:** complexity, churn, duplication, maintainability, smells, test coverage, hotspots.
* **Delta metrics:** before/after framework change.
* **Defect Leakage:**

  * defects per story;
  * defect origin vs discovery stage;
  * leakage rate (escaped ÷ total defects).
* **Rework & Stability:** code churn %, post-release fixes, human correction ratio.
* **Test Performance:** coverage delta, fail/pass ratio, mean time to fix.

### 16.3 Framework Effectiveness Ratios

To compare frameworks or setups (e.g., Dev-only vs Dev+Architect):

| KPI                         | Description                                                | Ideal Direction |
| --------------------------- | ---------------------------------------------------------- | --------------- |
| Token per LOC               | Total tokens ÷ lines of code generated                     | ↓               |
| Story time ratio            | Story duration ÷ total agent runtime                       | ↓               |
| Defect leakage              | Escaped defects ÷ total defects                            | ↓               |
| Code churn %                | Lines rewritten post-agent output                          | ↓               |
| Prompt count per feature    | Prompts to finish a story                                  | ↓               |
| Architect effectiveness     | (# refactors prevented or predicted defects) ÷ tokens used | ↑               |
| Cost per accepted LOC       | Tokens + time ÷ LOC merged                                 | ↓               |
| Agent coordination overhead | (non-code-agent tokens ÷ code-agent tokens)                | ↓               |

### 16.4 Comparability Across Complexity Bands

* All metrics normalized across complexity categories:

  * `business_complexity`
  * `technical_complexity`
  * `estimated_token_cost` (size/effort bucket)
* Comparisons always filtered by matching complexity bands to ensure apples-to-apples assessment.
* Each report includes segment-wise analysis (low, medium, high complexity) for accurate benchmarking.

### 16.5 Data Capture & Visualization

* Collected automatically in background workers via MCP events:

  * Tokens/time from `log_run`.
  * Code metrics from analyzer service.
  * Defects via `link_defect`.
  * Commits/tests via Git + CI integrations.
* Aggregated nightly and exposed through:

  * **Web UI dashboards:** efficiency, quality, framework comparison.
  * **MCP tools:** summarized KPIs for agents and planners.

## 17. Use Case Impact Tracking & Requirement Management

* During BA analysis:

  * System automatically lists all **use cases** impacted by the current story.
  * Each use case entry displays:

    * Linked requirements, screens, and flows.
    * Relationship type (modified / extended / deprecated).
  * Enables immediate visibility into which business requirements change per story.
* **Change tracking:**

  * When BA updates or creates use cases, the system versions them and logs diffs.
  * Every requirement change can spawn a new story automatically via a quick action (e.g., "Create Story from Requirement").
* **Logical structure:**

  * Use cases organized in an intuitive hierarchy:

    * By user flow, screen, or feature area.
    * Linked directly to stories, epics, and defects.
  * The structure allows easy navigation for non-technical users (PMs, stakeholders).
* **Review experience:**

  * Web UI displays requirements and use cases in expandable logical trees.
  * From each requirement node, user can:

    * View linked stories and defects.
    * Open documentation and history.
    * Initiate a new story to update that requirement.
* Ensures living, navigable business documentation always synchronized with technical execution.

## 18. Use-Case History & Full Traceability

* **Versioned use-cases:**

  * Every update creates a new version with:

    * `version_id`, `parent_use_case_id`.
    * author (human/agent), timestamp.
    * linked story/defect (if applicable).
    * machine-readable and human-readable diff summary.
* **Commit linkage:**

  * Mapping engine maintains relationships:

    * `use_case` ↔ `files` ↔ `commits`.
  * From a use-case:

    * View all commits that impacted its behavior.
  * From a commit:

    * View all use-cases potentially affected.
* **Defect linkage:**

  * Each defect (bug) includes `violated_use_cases[]` plus discovery stage and severity.
  * System can auto-suggest violated use-cases from changed files and history.
  * On defect fix:

    * Option to create a new use-case version documenting corrected behavior.
* **Regression protection:**

  * On new commits, system checks for impacted use-cases without matching story.
  * Flags "at-risk" use-cases until:

    * a story is created, or
    * change is explicitly reviewed.
* **UI support:**

  * Timeline view: requirement → stories → commits → defects → resolutions.
  * Diff view: compare any two use-case versions.
  * Quick navigation:

    * "Show defects for this use-case"
    * "Show commits that changed this use-case"
    * "Create story to adjust this requirement".
* Goal: never lose track of functionality or unintended behavioral changes.

## 19. Governance, Audit, and Release Management

### 19.1 Governance & Audit Layer

* **Immutable audit trail:**

  * Every change (stories, use-cases, requirements, commits, configs) recorded with author, timestamp, diff.
  * Stored in append-only log.
  * Linked directly to repository commits for traceability.
* **Repository synchronization:**

  * If repository is rolled back to a given commit, MCP system state also reverts to that commit's associated snapshot.
  * Enables full rollback of project documentation, stories, and metrics.
* **Versioned configuration:**

  * Frameworks, agents, and MCP settings versioned.
  * Rollback or diff comparison possible at any time.
* **Access control:**

  * Role-based permissions (Admin, PM, BA, Architect, Dev, QA, Viewer).
  * User identity tracked on every action.

### 19.2 Release & Delivery Management

* **Release entities:**

  * Group epics/stories into planned releases.
  * Each release has scope, start/end dates, environments, and associated commits.
* **Deployment linkage:**

  * Connect CI/CD pipelines and environment metadata.
  * Track when a release was deployed, rolled back, or patched.
* **Release metrics:**

  * Planned vs delivered scope.
  * Defects introduced per release.
  * Code churn and quality delta across releases.
* **Rollback tracking:**

  * When repo rollback occurs, release metadata and story/epic states revert in sync.
  * Preserve previous release artifacts and metrics for audit.

### 19.3 Security & Compliance

* **Secret redaction:**

  * All logs, prompts, and agent conversations automatically scrubbed for sensitive tokens, API keys, or PII.
* **Data residency:**

  * Configurable storage region for compliance.
* **Compliance reporting:**

  * Export audit logs and KPI summaries for ISO/SOC certification evidence.

### 19.4 Financial & Resource Forecasting

* **Token → cost mapping:**

  * Use provider rate tables to translate token use into monetary cost.
* **Budget dashboards:**

  * Show cumulative cost per project, story, or agent role.
* **Forecasting:**

  * Predict next-sprint usage and cost based on historical data.
  * Highlight outlier frameworks with high cost/benefit ratio.

## 21. MCP Tool API (Codex/Claude-ready)

Design tools as simple JSON-RPC methods over MCP.

### 21.1 Bootstrap & Sync

    * project_id
    * initial agents/frameworks
    * files[]: path + content for local config/skills.
* `get_changes({ project_id, since })`

  * Returns deltas:

    * changed stories, subtasks, use_cases, frameworks, artifacts.
    * new cursor.
